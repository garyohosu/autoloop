# AIで5日間開発したら「あと半月」と言われたので、AI開発そのものを自動化することにした

私は今、「OracleCouncil（神託会議）」という実験的なAIシステムを作っている。

複数のAIに同じ質問を考えさせ、回答の中から検証できる主張を取り出し、外部の情報を集め、最後に別のAIで監査してから回答を出す仕組みだ。

簡単に言えば、AIが思いつきで答えるのではなく、複数のAIと証拠を使って回答を点検するシステムである。

この開発にClaude CodeとCodex CLIを使い、5日ほど作業した。

昔のように私が一行ずつコードを書いているわけではない。設計資料を作り、AIへ実装を指示し、テストもAIに実行させている。

それでも、残り期間を見積もってもらうと、実験的なα版ならあと2～3日、他人が安心して使える状態まで考えると、さらに1週間から半月ほど必要かもしれないという。

すでに5日使っているのに、まだ半月かかる可能性がある。

そこで私は、OracleCouncilの続きを作る前に、**AI開発そのものを自動化する仕組み**を作ることにした。

それが「AutoLoop」である。

> 最新のインストール方法と使い方は、GitHubの[README.md](README.md)を参照してほしい。  
> この`note.md`は、noteへ掲載する記事原稿の正本である。

## AIにコードを書かせても、人間の仕事は残る

Claude CodeやCodex CLIに「実装して」と頼めば、かなりのところまでコードを書いてくれる。

しかし、実際の開発では、その前後に人間の作業が残る。

```text
AIの実装結果を読む
↓
テスト結果を確認する
↓
次に何を直すか決める
↓
新しい指示を書く
↓
もう一度AIを起動する
```

1回だけなら大した作業ではない。

ところが、機能追加、バグ修正、テスト失敗、未確定仕様が何十個も残っていると、この操作を何度も繰り返すことになる。

コードを書く部分はAIに任せられても、人間は画面の前で「次の指示を出す係」を続けなければならない。

OracleCouncilの残り期間を聞いたとき、問題はAIのコーディング速度だけではないと気づいた。

遅いのは、コード生成ではなく、開発の循環だった。

## コードを書くAIではなく、開発を繰り返す仕組み

最初に考えたAutoLoopは単純だった。

```text
設計資料と現在のコードを読む
↓
次の小さな作業を1つ選ぶ
↓
AIコーディングAgentを呼ぶ
↓
実装する
↓
テストする
↓
結果を保存する
↓
次の作業へ進む
```

PythonのControllerから、プロンプト付きでCodex CLI、Claude Code、Antigravityなどを非対話で呼び出す。

Agentが終了したら、固定した検証コマンドを実行し、標準出力、標準エラー、終了コード、変更ファイルをreceiptとして保存する。

プロジェクトが完成するか、本当に人間の判断が必要になるまで繰り返す。

私が欲しかったのは、巨大な自律開発基盤ではない。

**私は楽がしたい。**

結果を読み、次の指示を書き、AIを再起動する作業まで減らしたかった。

## 最初にできたのは、自動ループではなく単発Runnerだった

最初の実装では、人間が毎回`instructions.md`を用意し、Controllerがその内容をAgentへ渡す方式になった。

```text
人間がinstructions.mdを書く
↓
Agentを1回実行する
↓
result.mdを保存する
↓
終了
```

これは自動実行ではある。

しかし、次に何をするかを考えているのは、相変わらず人間だった。

私は「実装ボタン」を自動化したかったのではない。

「次に何を実装するか決める」「実装する」「確認する」「次へ進む」という開発ループを自動化したかった。

人間が毎回指示書を書かなければ動かないなら、それはAutoLoopではなく、起動ボタンをAIに置き換えただけである。

## Plannerを作ろうとしたら、今度は巨大になった

そこで、次の作業を選ぶPlannerを作ろうとした。

Plannerが設計資料、TODO、Git状態、前回のreceipt、テスト結果を読み、次のタスクを決める。

Controllerは指示書を生成し、Agentを起動し、変更範囲やテスト結果を検証し、stateとreceiptを保存する。

筋は通っているように見えた。

ところが独立レビューを行うと、次々に課題が出てきた。

- 変更可能パスをどう制限するか
- 既存の未コミットファイルをどう保護するか
- 検証コマンドを安全に実行する方法
- state破損時の扱い
- lockが残った場合の復旧
- timeout時の子孫プロセス管理
- renameや特殊なファイル名の扱い
- Agentの「完了しました」という自己申告をどう検証するか

どれも一般論としては正しい。

しかし、すべてを最初から実装すると、AutoLoopは個人用の小さな補助ツールではなく、自律開発プラットフォームになってしまう。

OracleCouncilを完成させるための道具を作っていたはずが、その道具を作るだけでトークンを使い果たしそうになった。

自分が楽をするためのソフトなのに、そのソフトの開発で疲れている。

これは何かがおかしい。

## 設計はspec-to-design、実装の反復はAutoLoop

整理してみると、設計工程はすでに別の仕組みで自動化できていた。

私は`spec-to-design`というスキルを使っている。

```text
SPEC.md
↓
USECASE.md
↓
SEQUENCE.md
↓
CLASS.md
↓
UI.md
↓
TESTCASE.md
```

各設計書は作成後に別のAIでレビューし、重大な矛盾や不足を直してから次へ進む。

つまり、AutoLoopに仕様策定や詳細設計まで担当させる必要はなかった。

役割を分ければよい。

```text
spec-to-design
    仕様から設計資料を作る

AutoLoop
    設計済みの項目を順番に実装する

Codex CLI / Claude Code / Antigravity
    実際のコード変更とテストを担当する

人間
    本当に人間でなければ決められないことだけ判断する
```

この形なら、Python側に巨大なPlannerを実装する必要はない。

設計資料、現在のコード、Git状態、前回の実行結果をAgentへ渡し、次の小さな実装項目をAgent自身に選ばせればよい。

## AutoLoopは小さなControllerへ戻った

現在のControllerの主な仕事は次のとおりである。

- JSON設定を読む
- 設計資料のファイル名をプロンプトへ追加する
- 設定されたAI Agentを起動する
- timeoutを監視する
- 固定した検証コマンドを実行する
- stdoutとstderrを保存する
- receiptを保存する
- 二重起動をlockで防ぐ
- 停止条件まで繰り返す

Python側に万能なTask Selectorは置かない。

次に何を実装するか、技術的な未確定事項をどう判断するかは、資料とコードを読めるAgentへ任せる。

AutoLoopは、賢いAIをPythonでもう一度作る仕組みではない。

**賢いAIを、決められた手順で繰り返し働かせる仕組み**である。

## 実際に未確定仕様の判断から実装まで進んだ

OracleCouncilでの実地試験では、AutoLoopが設計資料から未確定項目を見つけた。

最初は安全側に倒し、`HUMAN_CONFIRMATION`で停止した。

これは安全機能としては正しい。

しかし、技術的で後から変更でき、既存仕様やテストから妥当な案を選べる問題まで人間へ戻していては、私はあまり楽にならない。

そこでAgentへの指示を変えた。

未回答のQandAやBLOCKED項目を見つけても、すぐには停止しない。

関連する設計資料、コード、テスト、Git履歴を確認し、選択肢を比較する。

既存仕様との整合性、後方互換性、変更量、安全性、可逆性を基準に、最も妥当な技術的既定案を選ぶ。

決定内容は`QandA.md`へ`AUTO_DECIDED`として残す。

人間確認は、次のような問題に限定する。

- データ削除などの不可逆な変更
- 秘密情報や権限
- 金銭や有料API
- 外部公開やプライバシー
- 法務やライセンス
- 製品方針や好みに依存する判断
- 既存仕様を大きく壊す変更

次の試験では、AgentがOracleCouncilの未確定仕様S-9を見つけ、候補を比較し、最も妥当な案を選んだ。

その判断を設計資料へ記録し、コードとテストを変更し、全体で266件のテストを通過させた。

```text
未確定事項を発見
↓
Agentが資料とコードを調査
↓
安全な案を自動選択
↓
設計へ記録
↓
実装
↓
テスト
↓
receiptを保存
```

ここで初めて、私が作りたかったものが動いた。

## READMEをAI CLIへ渡すだけで導入できるようにした

AutoLoopはOracleCouncilの中だけで使うのではなく、独立したリポジトリとして公開した。

- GitHub: https://github.com/garyohosu/autoloop

AutoLoop本体と、AutoLoopを使う対象プロジェクトは分離して管理する。

対象プロジェクトのAI CLIへREADMEを読ませれば、必要な設定ファイルの生成、利用可能なAgentの選択、初回の1サイクル実行まで進められるようにした。

例えば、Antigravity、Codex CLI、Claude Codeなどへ次のように指示する。

```text
以下のREADMEを読んで、現在のGitリポジトリにAutoLoopを導入してください。

https://github.com/garyohosu/autoloop/blob/main/README.md

設定ファイルの生成、利用可能なAgentの選択、必要な事前確認を自動で行い、
最初は1サイクルだけ実行してください。

未コミットの変更を削除、stash、resetしないでください。
commitとpushも行わないでください。
最後に実行結果と停止理由を報告してください。
```

細かなインストール手順を人間が覚えるのではなく、READMEをAI CLI向けの実行仕様書として使う。

最新のコマンドや設定方法は、記事ではなくREADMEを正本としている。

## 最初は1サイクルだけ動かす

初回は`-Once`を指定して、1サイクルだけ実行する。

```powershell
.\.autoloop\run-autoloop.ps1 -Once
```

これでAgentの起動、ファイル変更、検証コマンド、ログ、receipt、停止理由を確認する。

最初から連続実行しない。

特に、未コミット変更が多い重要なリポジトリへ、いきなり自律ループを走らせるべきではない。

最初はクリーンなテスト用リポジトリやworktreeで動作確認した方がよい。

問題がなければ、`-Once`を外して複数サイクルを実行できる。

```powershell
.\.autoloop\run-autoloop.ps1
```

AutoLoop自身は`git commit`や`git push`を行わない。

実行ログとreceiptは対象プロジェクトの`.runtime`へ保存する。

## Windows版Codex CLIでもつまずいた

実地試験では、Codex CLIのWindowsサンドボックスが動かない問題も発生した。

専用ユーザーのSIDを解決できず、Windowsエラー1332で停止していた。

調査の結果、elevated構成は壊れたままだったが、`[windows] sandbox = "unelevated"`へ設定することで通常運用は復旧した。

また、CLIの構文を誤って`codex sandbox windows ...`のように実行すると、存在しない`windows.exe`を起動しようとして別のエラーになることも分かった。

こういう問題は、机上で安全機能を考えているだけでは見つからない。

最小版を実際に動かし、発生した問題だけを直す方が早い。

## Agentを交換しても続けられる形にしたい

開発中、Claude Codeが利用上限へ達し、Codex CLIにもrate limitの警告が出た。

そこでAntigravityも利用候補になった。

AutoLoopでは、使用するAgentをPythonコードへ固定せず、設定ファイルのコマンドとして扱っている。

そのため、同じ設計資料、Git状態、テスト結果、receiptを別のAgentへ読ませられる。

会話履歴そのものを引き継ぐのではない。

正本となるファイルを共有する。

- Gitリポジトリ
- 設計資料
- QandA
- テスト結果
- receipt
- stdoutとstderr

Agentが変わっても、これらを読めば現在位置を再現しやすい。

現時点では、利用上限を検出して別Agentへ自動切替する機能までは実装していない。

まずは設定を切り替えて再開できれば十分である。

## 現在できること

現在のPhase 1では、主に次のことができる。

- Antigravity、Codex CLI、Claude Code向け設定の生成
- READMEを読んだAI CLIによる自動導入
- 1サイクルまたは複数サイクルの実行
- Agentによる次の小さな作業の選択
- 固定verificationコマンドの実行
- stdout、stderr、receiptの保存
- 二重起動防止
- 技術的で可逆的な未確定事項の自動判断

一方、まだ次の機能はない。

- 利用上限時のAgent自動切替
- セッションのresume
- 子孫プロセスの完全な管理
- 自動commit・push
- Mac/Linux用のインストールスクリプト
- Web UI

万能な自律開発基盤ではない。

設計済みのプロジェクトで、AIコーディングAgentを繰り返し働かせるための小さな個人用ツールである。

## これは「ループエンジニアリング」かもしれない

AIコーディングでは、プロンプトを書いてコードを生成する部分に注目が集まりやすい。

しかし、実際の開発には、その前後がある。

```text
計画する
実装する
検証する
結果を記録する
次を決める
```

この循環をどのように作るか。

どこまでAIに判断させ、どこで人間へ戻すか。

失敗した結果を次のサイクルへどう渡すか。

Agentを賢くするだけでなく、Agentが働き続けられるループを設計する。

これを「ループエンジニアリング」と呼んでもよいかもしれない。

重要なのは、巨大で万能な自律開発基盤を作ることではない。

自分の作業で繰り返している部分を見つけ、AIが続けられる形へ変えることである。

## 今回の教訓

### コード生成だけでは開発は自動化されない

AIがコードを書いても、人間が結果を読み、次の指示を書き続けるなら、人間の待ち時間は減らない。

自動化すべきなのは、コードだけでなく開発の循環である。

### 設計と実装の役割を分ける

設計工程は`spec-to-design`で完了させる。

AutoLoopは設計済みの項目を実装する。

何でも1つのシステムへ詰め込まない。

### AIが判断できることまで人間へ戻さない

技術的で可逆的な判断はAgentへ任せる。

何でも勝手に決めさせるのではなく、自動決定してよい境界を明確にする。

### 最初から完璧な基盤を作らない

実際に使い、困った機能だけ追加する。

ツールを作ること自体が新しい大仕事になったら、一度、最初の目的へ戻った方がよい。

### 自分が楽になるかを基準にする

安全装置を増やした結果、人間確認が増えたら意味がない。

安全な範囲ではAIが進み、本当に必要な場面だけ人間へ戻す。

## これから

OracleCouncilを作っていたはずが、途中から「OracleCouncilを作り続ける仕組み」を作ることになった。

遠回りにはなった。

しかし、AutoLoopの開発に何日かかったかが、この話の中心ではない。

出発点は、OracleCouncilをAIで5日間開発しても、まだ2日から半月ほどかかると言われたことだった。

AIにコードを書かせるだけでは、人間の仕事はまだ多い。

次に自動化すべきなのは、コードではなく、その前後にある開発の循環なのかもしれない。
